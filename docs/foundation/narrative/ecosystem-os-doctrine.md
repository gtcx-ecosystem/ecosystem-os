# Ecosystem OS Doctrine

## Product thesis

Ecosystem OS is the integrated operating system for running an AI-native company or ecosystem.

It is not a documentation repo, a sprint board, an agent framework, or an infrastructure toolkit by itself. It is the operating layer that connects company intelligence, human judgment, agent coordination, delivery, services, assurance, and market expression into one governed system.

## Boundary

| System | Role |
| ------ | ---- |
| **Ecosystem OS** | Operates the company and ecosystem. |
| **Canon OS** | Publishes official documentation and knowledge surfaces. |
| **GTCX OS** | Builds the trade platform and product implementation. |
| **Baseline OS** | Provides AI/runtime substrate and agent runtime capabilities. |

## Core claim

AI-native companies need more than copilots. They need an operating system where humans and agents can:

1. Know the state of the company.
2. Exercise judgment through explicit doctrine and authority.
3. Invent products, IP, and strategic bets.
4. Express the company through creative, comms, marketing, and brand.
5. Coordinate work across people, agents, tools, and repos.
6. Plan, deliver, and ship.
7. Operate infrastructure, services, and cross-functional Ops.
8. Verify quality, compliance, risk, and evidence.
9. Publish the official surface through Canon OS.

## Operating loop

```text
Know -> Judge -> Invent -> Express -> Coordinate -> Ship -> Operate -> Verify -> Publish -> Learn
```

Learning feeds back into Graph and Ledger. Canon publishes selected outputs. Assurance determines whether evidence is sufficient. Bridge routes humans, agents, and tools through the operating model.

## Design principle

Repo boundaries are implementation details. Product boundaries are conceptual operating capabilities.

Ecosystem OS should be designed as a portfolio of platform products that can run internally, expose APIs, support agents through MCP, support RAG through Graph, and eventually be packaged as open-source or commercial modules.
