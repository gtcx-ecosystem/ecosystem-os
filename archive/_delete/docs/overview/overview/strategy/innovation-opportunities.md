# Innovation Opportunities — gtcx-agile

**Date:** 2026-03-09
**Scope:** ai-1-agile — what better coordination infrastructure unlocks; AI-native planning patterns

---

## The Core Opportunity

ai-1-agile is unique in the ecosystem: it has no feature backlog, no users, and no product to ship. Its output is coordination quality. Better coordination does not create one improvement — it compounds across every repo it touches. A 10% improvement in cross-repo coordination quality translates to a materially shorter critical path to Ghana.

The innovations here are not technical features. They are coordination patterns that do not currently exist.

---

## Opportunity 1: Living Cross-Repo Dependency Graph

### Current State

Cross-repo dependencies exist (six platforms depend on protocols, frontend depends on platforms) but are invisible. They are not tracked, not dated, not owned.

### The Innovation

A dependency graph that is a first-class artifact — updated every sprint, not as a diagram but as a structured table that feeds into sprint planning:

```markdown
| Dependency            | From Repo      | To Repo        | What                               | Needed By  | Status           |
| --------------------- | -------------- | -------------- | ---------------------------------- | ---------- | ---------------- |
| AGX MVP API           | 6-platforms    | ai-3-fiftyfour | /api/trade/submit endpoint         | 2026-04-15 | Not started      |
| TradePass integration | 3-protocols    | 6-platforms    | @gtcx/protocol-tradepass consumed  | 2026-04-01 | Ready to consume |
| Cortex service        | 5-intelligence | 6-platforms    | gRPC endpoint live                 | 2026-05-01 | No timeline      |
| Land Credit Score API | ai-6-terra     | ai-3-fiftyfour | /api/v2/institutional/credit-score | 2026-04-30 | Backend complete |
```

This table should be reviewed in every ecosystem sprint kickoff. Any row without a "Needed By" date or with "No timeline" status is an escalation trigger.

### Why This Matters for Q2

The Ghana demo requires a chain of five systems working in sequence. Right now no one is watching the chain. This table would make every link visible.

## Opportunity 2: Agentic Sprint Planning

### Current State

Sprint planning is manual and ad-hoc. There are no active sprints documented in ai-1-agile. No shared sprint cadence exists across repos.

### The Innovation

An agent-assisted sprint kickoff that:

1. Reads `git log --since=2weeks` across all active repos
2. Identifies which planned P0 items shipped, which slipped, and what is newly unblocked
3. Drafts the next sprint plan for each repo with dependency handoffs pre-populated
4. Surfaces any repo where velocity has dropped below a threshold (e.g., no commits in 7 days)

This is achievable with the existing GTCX agent infrastructure (1-agentic repo) and would take less than a sprint to build. The output is a draft sprint doc that a human reviews and finalizes — not autonomous planning, but AI-accelerated preparation.

**Concrete output per sprint kickoff:**

- `_sop/3-agile/2-scrum-board/5-sprints/sprint-\{n\}-\{date\}.md` — pre-populated with repo status, dependency deltas, and suggested P0 items
- A diff against last sprint's plan showing what completed, what slipped, and what is new

## Opportunity 3: Slip Detection and Escalation Automation

### Current State

When a dependency slips (e.g., 6-platforms misses its AGX MVP date), the downstream repos (ai-3-fiftyfour) do not know unless they are watching. This compounds silently.

### The Innovation

A weekly automated slip check:

1. For each entry in the dependency register, check the "Needed By" date against current status
2. If a dependency is overdue or unstarted 2 weeks before its date, flag it
3. Generate a slip report as a markdown file in `_sop/3-agile/4-audits/`
4. Optionally notify via a configured webhook

This is a scripts task. The `03-platform/scripts/` directory exists. The logic is simple: a YAML or markdown dependency register plus a Python script that checks dates and outputs flags. Low effort, high value.

## Opportunity 4: Velocity Dashboard as a Static Site

### Current State

There is no velocity tracking. Commit cadence per repo is visible only to those who run `git log` manually.

### The Innovation

A static dashboard (can be a single HTML file generated by a script) that shows:

- Commits per repo in the last 30 days
- Last commit date per repo
- Repo tier (shipping / foundation / spec / early) as a status indicator
- Open dependency count per repo

Published to GitHub Pages or as a build artifact. Updated on a cron. Zero infrastructure beyond what already exists.

This makes ecosystem health visible to everyone — engineers, executives, DFI partners — without requiring access to each repo.

## Opportunity 5: Architecture Decision Record (ADR) as a Coordination Primitive

### Current State

Prioritization decisions happen but are not recorded. When a decision is made ("build AGX before mobile"), there is no artifact that captures it, and the same decision gets re-litigated.

### The Innovation

Every significant cross-repo prioritization decision becomes an ADR in `_sop/reference/archive/legacy-authoritative/_archive/decisions/`. The format is simple:

```markdown
# ADR-XXX: Build 6-platforms AGX before 7-mobile

**Status:** Accepted
**Date:** 2026-03-09
**Decision:** Prioritize AGX MVP (6-platforms) over mobile feature implementation (7-mobile) in Q2.
**Rationale:** AGX unblocks 13 product surfaces in fifty-four. Mobile unblocks field testing.
AGX has greater Q2 Ghana revenue impact. Mobile phase starts Q3.
**Consequence:** 7-mobile begins feature implementation no earlier than 2026-06-01.
```

This is not process overhead. It is preventing the same prioritization conversation from happening three times at different decision points.

## Opportunity 6: Q2 Retrospective Template Designed for a 15-Repo Ecosystem

### Current State

Retrospective templates in the scrum board folder are single-product templates. They do not account for cross-repo failures (e.g., "platform B slipped because it was waiting for platform A").

### The Innovation

An ecosystem retrospective format that captures:

- Per-repo: what shipped, what slipped, what was blocked externally vs. internally
- Cross-repo: which dependency handoffs failed, which slips cascaded, what coordination could have caught it earlier
- Process: what the coordination layer (this repo) failed to surface in time

Run once at the end of Q2. Output goes to `_sop/3-agile/2-scrum-board/7-retrospectives/`. This builds the institutional memory that prevents the same coordination failures in Q3.

## What Better Coordination Unlocks Across the Ecosystem

The innovation opportunities above are not incremental improvements. They unlock:

| If This Exists          | Then This Becomes Possible                           |
| ----------------------- | ---------------------------------------------------- |
| Dependency register     | Slip detection before it cascades                    |
| Agentic sprint kickoff  | Sprint planning takes hours, not days                |
| Velocity dashboard      | Stalled repos are visible in real time               |
| ADR record              | Prioritization decisions are durable and retrievable |
| Ecosystem retrospective | Q3 planning starts from learned Q2 patterns          |

**The compounding effect:** A coordination layer that actually works means 14 other repos build faster, with less rework, and with visible risk before it becomes a schedule failure.

---

## The One Innovation That Would Have the Most Impact

Of everything above: the **living dependency register** with weekly slip detection.

It does not require AI. It does not require new infrastructure. It requires a markdown table and a script. But it would have caught the frontend-ahead-of-backends mismatch months ago, and it would surface today that the 6-platforms build is unscheduled while thirteen product surfaces in ai-3-fiftyfour are waiting for it.

That single artifact, maintained consistently, would change the coordination quality of the entire 15-repo ecosystem.
